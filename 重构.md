# 五层架构重构文档

> 记录从旧三模块架构到新五层架构的完整重构过程。
> 包含背景分析、设计决策、文件迁移、依赖关系、验证方法。

---

## 一、重构背景

### 1.1 项目概述

本项目是一个面向四足机器人（Unitree Go2 / IsaacLab 仿真）的 LLM Agent 系统。用户通过自然语言下达任务，系统通过大模型规划、参数计算、技能执行、状态验收完成闭环。

核心链路：

```
用户输入 → LLM 判断意图 → vlm_observe(观察) / robot_act(动作) → 规划 → 执行 → 验收
```

### 1.2 旧架构

重构前项目包含 5 个模块：

```
Comm_Module/       通信与状态层
VLM_Module/        视觉语言模型层
Agent_Module/      规划与执行代理层
Excu_Module/       执行层
Robot_Module/      技能注册与实现层
```

### 1.3 旧架构的问题

#### 职责混乱

`Agent_Module` 同时承担了：
- TUI 交互界面（`replanner.py` 中的 `InteractiveSessionState`、`main()`）
- 高层任务规划（`planner.py` 中的 CoT 推理链）
- 参数计算（`parameter.py` 中的 `ParameterCalculator`）
- 执行流水线（`replanner.py` 中的 `PipelineRunner`）
- 低层任务执行（`executor.py` 中的 `LowLevelExecutor`）

这些职责之间没有清晰的边界，修改任何一个都可能影响其他功能。

#### 循环依赖

旧架构中存在多处不合理的依赖关系：

```
Agent_Module → Robot_Module → Excu_Module → Comm_Module
     ↑                                           ↑
     └───────────────────────────────────────────┘
```

`Agent_Module/replanner.py` 中的 `PipelineRunner` 直接导入 `Robot_Module` 和 `Excu_Module`，
而 `Robot_Module/agent_tools.py` 又导入 `Agent_Module/planner.py`。
这种双向依赖导致模块无法独立测试和替换。

#### 通信层耦合业务逻辑

`Comm_Module/Task/__init__.py` 中混合了：
- 后端注册与管理（`TaskBackend`、`register_task()`）
- 数据同步（`sync_object_facts_from_live_data()`、`sync_runtime_overrides_from_user_input()`）
- 状态获取（`get_task_data()`、`get_task_state()`）

数据同步是业务逻辑，不应该放在通信抽象层中。

#### 换项目成本高

如果要将项目从 Bishe 任务迁移到其他任务类型，需要修改：
- `Comm_Module`（添加新后端）
- `Agent_Module/parameter.py`（修改参数计算）
- `Robot_Module/module/Action/Task/Bishe/`（替换技能实现）
- `Agent_Module/planner.py`（可能需要修改规则覆盖）

理想情况下，换项目只需要修改 `Robot_Module` 和 `Hardware_Module`。

#### VLM 与数据获取耦合

`VLM_Module` 只负责视觉感知，但 `Agent_Module/parameter.py` 同时包含：
- `load_object_facts()` — 从 JSON 文件加载物体事实
- `normalize_object_facts()` — 标准化物体事实
- `ParameterCalculator` — 根据物体几何计算参数

物体事实加载和参数计算属于"数据感知"范畴，不应该和"规划代理"混在一起。

## 二、新架构设计

### 2.1 设计原则

1. **单一职责** — 每个模块只有一个变更原因
2. **依赖倒置** — 上层定义接口，下层提供实现
3. **依赖注入** — 通过构造函数或回调注入依赖，而非硬编码导入
4. **可插拔后端** — 硬件后端通过注册表管理，运行时可切换
5. **项目无关** — 通用模块不包含任何项目特定代码

### 2.2 五层架构

```
┌─────────────────────────────────────────────────────────────┐
│  cli.py / run.py                                            │
│  TUI 交互界面入口                                            │
├─────────────────────────────────────────────────────────────┤
│  Robot_Module — 技能注册层                                   │
│  MCP 工具注册 + 任务分发 + 具体技能实现                       │
│  包含项目特定代码（Bishe）                                    │
├─────────────────────────────────────────────────────────────┤
│  Excu_Module — 执行流水线层                                  │
│  pipeline 编排 + 统一执行 + 等待反馈 + 状态校验              │
│  不包含项目特定代码                                          │
├─────────────────────────────────────────────────────────────┤
│  Planner_Module — LLM 规划层                                │
│  高层 CoT 规划 + 规则覆盖 + 低层 LLM 工具选择               │
│  通过依赖注入和回调工作                                      │
├─────────────────────────────────────────────────────────────┤
│  Data_Module — 数据感知层                                    │
│  VLM 视觉语义 + object_facts + 参数计算 + 上下文构建         │
├─────────────────────────────────────────────────────────────┤
│  Hardware_Module — 硬件抽象层                                │
│  统一状态入口 + 后端注册 + sim/Go2 后端                      │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 依赖关系（单向）

新架构严格保证依赖方向从上到下，不存在循环依赖：

```
cli.py
  → Robot_Module
    → Excu_Module
      → Hardware_Module
    → Planner_Module (无外部依赖)
    → Data_Module
      → Hardware_Module
    → Hardware_Module

Planner_Module    → (无外部模块依赖，通过依赖注入)
Excu_Module       → Hardware_Module
Data_Module       → Hardware_Module
Hardware_Module   → (无外部模块依赖)
```

### 2.4 换项目影响范围

```
换项目只需修改：
  ✏ Robot_Module/tasks/<new_project>/   ← 新技能实现
  ✏ Robot_Module/tasks/__init__.py      ← 注册新任务类型
  ✏ Hardware_Module/backends/<new>/     ← 新硬件后端（可选）

零修改：
  ✅ Hardware_Module/registry.py         ← 通用后端管理
  ✅ Hardware_Module/schema.py           ← 通用状态解析
  ✅ Data_Module/                        ← VLM + facts + params
  ✅ Planner_Module/                     ← LLM 规划 + 规则
  ✅ Excu_Module/                        ← 执行 + 验收
```

#### 关键设计决策

**延迟导入（Lazy Import）**

`backends/sim/__init__.py` 和 `backends/go2/__init__.py` 使用 `__getattr__` 实现延迟导入：

```python
_EXPORTS = {
    "STATE_SCHEMA": "Hardware_Module.backends.sim.schema",
    "TASK_DATA": "Hardware_Module.backends.sim.schema",
    "get_data": "Hardware_Module.backends.sim.data",
    ...
}

def __getattr__(name: str):
    module_path = _EXPORTS.get(name)
    if module_path is None:
        raise AttributeError(f"module {__name__!r} has no attribute {name!r}")
    module = import_module(module_path)
    return getattr(module, name)
```

这样做的好处：
- 不使用 Go2 后端时，不会触发 ROS2 导入（避免 NumPy 兼容性问题）
- 不使用 Sim 后端时，不会读仿真文件
- 按需加载，减少启动时间

**状态解析引擎**

`schema.py` 中的 `_parse_schema()` 是一个递归状态解析器：

```python
STATE_SCHEMA = {
    "connected": {"source": "connected", "default": False},
    "observation": {
        "agent_position": {"source": "snapshot.robot_pose", "default": None},
    },
}
```

- `source` — 从原始数据的嵌套路径取值（支持 `.` 分隔）
- `default` — 取不到值时的默认值
- `literal` — 固定值（不从原始数据取）

每个后端定义自己的 `STATE_SCHEMA`，`schema.py` 的 `get_state()` 统一解析：

```python
def get_state(task_type=None):
    backend = load_task_backend(task_type)       # 加载后端定义
    raw_data = get_task_data(backend.task_type)  # 获取原始数据
    state = _parse_schema(backend.state_schema, raw_data)  # 统一解析
    return state
```

**后端注册表**

`registry.py` 管理 `TaskBackend` 注册和加载：

```python
@dataclass(frozen=True)
class TaskBackend:
    task_type: str
    description: str
    data_module: str                                    # 数据获取模块路径
    state_schema: dict[str, Any]                        # 状态格式定义
    data_getter_name: str = "get_data"                  # 数据获取函数名
    sync_object_facts_from_live_data_name: str | None   # 同步函数名
    sync_runtime_overrides_from_user_input_name: str | None
```

通过 `register_task()` 注册，`load_task_backend()` 加载。当前内置 `sim` 和 `go2` 两个后端。

通过环境变量切换后端：

```bash
export FINALPROJECT_ROBOT_TYPE=sim   # 默认
export FINALPROJECT_ROBOT_TYPE=go2   # 真机
```

#### 关键设计决策

**依赖注入取代直接导入**

旧代码中 `Planner` 直接导入 `ParameterCalculator`：

```python
# 旧 Agent_Module/planner.py
from Agent_Module.parameter import ParameterCalculator

class Planner:
    def annotate_tasks(self, tasks, object_facts):
        calculator = ParameterCalculator()
        return calculator.annotate_tasks(tasks, object_facts)
```

新代码通过构造函数注入：

```python
# 新 Planner_Module/planner.py
class Planner:
    def __init__(self, client, model=None, prompt_path=None, parameter_calculator=None):
        self.parameter_calculator = parameter_calculator  # 注入

    def annotate_tasks(self, tasks, object_facts):
        if not object_facts or self.parameter_calculator is None:
            return tasks
        return self.parameter_calculator.annotate_tasks(tasks, object_facts)
```

好处：
- `Planner` 不再依赖 `Data_Module`
- 可以不传 `parameter_calculator`，此时跳过参数计算
- 单元测试时可以传入 mock 对象

**直接执行 planner 输出**

当前版本已取消 `Planner_Module/executor.py` / `TaskExecutor`。高层 Planner 直接输出 `function` + `params`，`Excu_Module/pipeline.py` 对每个任务调用注入的 `execute_tool_fn`，再由 `Robot_Module/tool_runtime.py` 找到对应技能函数执行。

好处：
- 少一层低层 LLM 工具选择，链路更短
- 文档、日志和真实调用路径一致
- 工具执行仍通过回调注入，`Planner_Module` 不导入 `Robot_Module`

**规则覆盖拆出**

两种规则覆盖逻辑（box-assisted 和 climbable-obstacle）已拆到 `Planner_Module/rule_overrides.py`，因为它们是独立的场景几何判断，不应该塞在 `Planner` 主类里：

```python
def apply_rule_overrides(tasks, meta, *, user_input, scene_facts, object_facts):
    # 1. box-assisted: 两侧高台阻挡 + 可移动箱子 → push_box -> climb_align -> climb -> nav_climb
    geometry = self._detect_box_assisted_geometry(scene_facts, object_facts)
    if geometry and self._should_override_box_assisted_plan(tasks, user_input):
        return self._build_box_assisted_plan(geometry)

    # 2. climbable-obstacle: 两侧平台 + 一侧可攀爬 → way_select -> nav_climb
    climbable = self._detect_climbable_obstacle(scene_facts, object_facts)
    if climbable and self._is_navigation_request(user_input):
        return self._build_climbable_obstacle_plan(climbable)

    return tasks, meta
```

**向后兼容别名**

## 六、设计模式

### 6.1 依赖注入（Dependency Injection）

**问题**：`Planner` 直接创建 `ParameterCalculator` 实例，导致 `Planner_Module` 依赖 `Data_Module`。

**解决方案**：通过构造函数注入。

```python
# Robot_Module/tools.py 中创建 Planner 时注入
from Data_Module.params import ParameterCalculator
from Planner_Module.planner import Planner

parameter_calculator = ParameterCalculator()
planner = Planner(client=client, parameter_calculator=parameter_calculator)
```

依赖方向：
```
Robot_Module → Planner_Module (创建时注入 Data_Module.params)
            → Data_Module (提供 ParameterCalculator)
```
而不是：
```
Planner_Module → Data_Module (硬编码导入) ← 旧行为，已消除
```

### 6.2 回调模式（Callback Pattern）

**问题**：旧执行器直接调用 `Robot_Module` 的工具函数，导致双向依赖。

**解决方案**：`Excu_Module/pipeline.py` 只接收 `execute_tool_fn` 回调，逐步执行 Planner 输出的 `function` + `params`。

```python
# Excu_Module/pipeline.py
def run_pipeline(user_input, planner, execute_tool_fn, ...):
    for task in tasks:
        result = execute_tool_fn(task["function"], task["params"])
```

```python
# Robot_Module/tools.py 中提供回调实现
def _execute_registered_tool(function_name, function_args):
    skill_func = get_skill_function(function_name)
    raw_result = _run_async_blocking(skill_func(**function_args))
    return _normalize_tool_result(function_name, raw_result)
```

### 6.3 策略模式（Strategy Pattern）

**问题**：不同硬件后端（sim/go2）有不同的数据获取方式。

**解决方案**：通过 `TaskBackend` 注册表实现策略切换。

```python
# Hardware_Module/registry.py
@dataclass(frozen=True)
class TaskBackend:
    task_type: str
    data_module: str                    # 策略模块路径
    state_schema: dict                  # 状态格式
    data_getter_name: str = "get_data"  # 策略函数名

# 运行时根据 FINALPROJECT_ROBOT_TYPE 选择策略
backend = load_task_backend()  # 自动选择 sim 或 go2
raw_data = get_task_data()     # 调用对应后端的 get_data()
state = _parse_schema(backend.state_schema, raw_data)
```

### 6.4 延迟导入（Lazy Import）

**问题**：导入 Go2 后端会触发 ROS2 导入，在不使用 ROS2 的环境下报错。

**解决方案**：使用 `__getattr__` 实现模块级延迟导入。

```python
# Hardware_Module/backends/go2/__init__.py
_EXPORTS = {
    "get_data": "Hardware_Module.backends.go2.data",
    "STATE_SCHEMA": "Hardware_Module.backends.go2.schema",
    ...
}

def __getattr__(name: str):
    module_path = _EXPORTS.get(name)
    if module_path is None:
        raise AttributeError(...)
    module = import_module(module_path)
    return getattr(module, name)
```

只有真正访问 `Hardware_Module.backends.go2.get_data` 时才会导入 ROS2 相关模块。

### 6.5 模板方法模式（Template Method）

**问题**：7 个技能文件有相似但不同的执行流程。

**解决方案**：`SkillBase` 基类定义模板方法，具体技能实现细节。

```python
# Excu_Module/skill_base.py
class SkillBase:
    name: str

    def execute(self, **kwargs):
        """模板方法：参数准备 → 命令下发 → 等待反馈 → 返回结果"""
        ...

    def check_completion(self, state):
        """子类实现：轮询完成判定"""
        ...

    def validate(self, pre_state, post_state):
        """子类实现：事后校验"""
        ...
```

每个技能文件（如 `walk.py`、`navigation.py`）继承 `SkillBase` 并实现具体逻辑。

## 十四、总结

本次重构的核心变化：

1. **消除循环依赖** — 通过依赖注入和回调模式，所有模块依赖方向从上到下
2. **职责分离** — 5 个模块各司其职，Agent_Module 的混乱职责被拆散
3. **可插拔后端** — 硬件后端通过注册表管理，运行时可切换
4. **项目无关** — 通用模块（Data/Planner/Excu）不包含任何 Bishe 代码
5. **扁平化结构** — Robot_Module 从 `module/Action/Task/Bishe/` 5 层嵌套变为 `tasks/bishe/` 2 层

重构后的架构更清晰、更易维护、更易扩展。

## 十五、Rich TUI 重设计

> 在五层架构重构完成后，对 TUI 交互界面进行了全面重设计。
> 引入 `on_event` 回调机制，统一 Rich 渲染，增加实时状态展示。

### 15.1 问题分析

五层架构重构后，`cli.py` 直接继承了旧 `Agent_Module/replanner.py` 的 TUI 代码，存在三个问题：

1. **缺少实时状态** — 无机器人位置、连接状态等实时信息展示
2. **日志和面板混杂** — `pipeline.py` / `planner.py` 的 `print()` 与 Rich Panel 交错
3. **信息展示混乱** — 欢迎面板、状态行、对话流组织不佳

### 15.2 on_event 回调机制

**核心思路：** 将 `pipeline.py` 和 `planner.py` 中的所有裸 `print()` 替换为结构化事件回调，由 TUI 层控制渲染。

**类型定义：**

```python
# Excu_Module/pipeline.py
from typing import Any, Callable

PipelineEventCallback = Callable[[str, dict[str, Any]], None]
```

**事件类型：**

| 事件 | 数据 | 说明 |
|------|------|------|
| `plan_start` | `{user_input}` | 开始规划 |
| `plan_thinking` | `{thinking}` | LLM 推理过程 |
| `plan_done` | `{tasks, summary}` | 规划完成 |
| `plan_error` | `{error}` | 规划失败 |
| `rule_override` | `{reason, tasks}` | 规则覆盖 |
| `execute_start` | `{}` | 开始执行 |
| `step_start` | `{idx, total, function, params, reason}` | 步骤开始 |
| `step_progress` | `{pose}` | 实时坐标（每秒） |
| `step_done` | `{idx, success, message}` | 步骤完成 |
| `replan` | `{message}` | 重规划 |
| `pipeline_done` | `{results}` | 流水线完成 |
| `pipeline_error` | `{error}` | 流水线错误 |

**降级机制：** 当 `on_event is None` 时，所有事件回退为原来的 `print()` 行为，保持向后兼容。

### 15.3 修改的文件

#### `Excu_Module/pipeline.py`

- 新增 `_execute_with_progress()` — 后台线程执行技能 + 主线程每秒轮询机器人坐标

```python
def _execute_with_progress(
    execute_tool_fn: Callable,
    function_name: str,
    params: dict,
    on_event: PipelineEventCallback,
) -> dict[str, Any]:
    """后台线程执行技能，主线程每秒轮询坐标回传。"""
    result_box: dict[str, Any] = {}
    error_box: dict[str, BaseException] = {}

    def _worker():
        try:
            result_box["value"] = execute_tool_fn(function_name, params)
        except BaseException as exc:
            error_box["value"] = exc

    thread = threading.Thread(target=_worker, daemon=True)
    thread.start()

    while thread.is_alive():
        thread.join(timeout=_PROGRESS_POLL_SEC)  # 1.0 秒
        if thread.is_alive():
            try:
                from Hardware_Module import get_state
                from Excu_Module.state import summarize_state
                state = get_state()
                if state and isinstance(state, dict):
                    summary = summarize_state(state) or {}
                    on_event("step_progress", {"pose": summary.get("robot_pose")})
            except Exception:
                pass

    if "value" in error_box:
        raise error_box["value"]
    return result_box.get("value", {})
```

### 15.5 位移方向宽限期

**问题：** 技能切换时（如"往前走1米再往后走2米"），机器人惯性导致下一步的第一个轮询采样到上一步的位移方向，触发方向校验失败。

**解决方案：** 在 `Excu_Module/state.py` 的 `wait_for_displacement_completion()` 中添加方向宽限期：

```python
DEFAULT_DIRECTION_GRACE_SEC = 1.0  # 可通过 FINALPROJECT_DIRECTION_GRACE_SEC 环境变量配置
grace_deadline = time.time() + direction_grace_sec

# 宽限期内：跳过方向检查
# 宽限期结束：用当前位置重新校准 start_pose，消除惯性漂移
in_grace = time.time() < grace_deadline
if not in_grace and effective_start_pose == list(start_pose):
    effective_start_pose = list(robot_pose)
```

### 15.6 walk 四方向支持

**问题：** `walk.py` 只处理前进/后退（vx），左右方向（vy）默认落到前进分支。

**修复：** 添加左/右方向处理：

| route_side | velocity | 说明 |
|------------|----------|------|
| 前 | `[+vx, 0, 0]` | 前进 |
| 后 | `[-vx, 0, 0]` | 后退 |
| 左 | `[0, -vy, 0]` | 左移 |
| 右 | `[0, +vy, 0]` | 右移 |

对应 ROS2 `cmd_vel`：`twist.linear.y` 需要支持非零值。

### 15.8 walk 超时公式修正

**问题：** 机器人实际速度（~0.18 m/s）远低于配置速度（0.6 m/s），原公式 `timeout + 2.0` 导致 1m 行走只有 4.2s，不足。

**修复：**

```python
# 旧
timeout_sec = estimate_linear_duration(distance, speed) + 2.0

# 新
timeout_sec = max(timeout * 3, timeout + 8.0)
```

1m 行走：`max(3 * 2.17, 2.17 + 8.0) = max(6.51, 10.17) = 10.17s`
